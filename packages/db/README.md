# @sancta/db — migrations & schema

Forward-only, versioned SQL migrations applied **identically** to the clinic edge
PostgreSQL and the managed cloud PostgreSQL (ADR-0001/0002). One schema, two runtimes.

- `migrations/0001_init.sql` — canonical baseline: domain schemas (identity, flow,
  clinical, billing, inventory, finance) plus the cross-cutting `security_sync` and
  `audit` tables every domain depends on.

## Invariants encoded in the schema

- **UUIDv7 primary keys** generated by the application offline (ADR-0006).
- **Universal fields** (provenance, versioning, sync, security) on business tables (§14.1).
- **Balances are derived, never stored**: `inventory.stock_balance` is a view over the
  append-only `inventory.stock_movement`; there is no editable on-hand column (BR-007).
- **Journals**: `finance.journal_line` forbids negative amounts and a line that is both
  debit and credit; balancing (Σ debit = Σ credit) is enforced by `@sancta/domain`
  `assertPostable()` before insert and re-checked at sync ingress. System batches are
  reversed via a linked `reverses` batch, never edited (BR-009).
- **Idempotency**: `security_sync.applied_change` has the idempotency key as its primary
  key, so a duplicate delivery is a no-op insert — the DB guarantee behind
  `IdempotentApplier` (NFR-010).
- **Append-only audit**: `audit.audit_event` carries a hash chain (`prev_hash`/`event_hash`)
  for tamper evidence (NFR-016); application code never UPDATEs or DELETEs it.

## Migration tooling (Phase 1)

A migration runner compatible with Node.js (edge, direct `pg`) and Cloudflare Workers
(cloud, via Hyperdrive) applies these files in order and records applied versions. Rollback
scripts accompany each forward migration (NFR-024). No patient data is ever seeded here —
see `/migration` for the workbook cut-over and `/seed` for synthetic data.
